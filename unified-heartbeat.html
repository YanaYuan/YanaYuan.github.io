Unified Heartbeat

你的跨项目自主执行循环——自动扫描所有项目、收集信息、思考决策、执行任务，每一步都有迹可循。

concordia-pmo-manager / skills / unified-heartbeat

💡这是什么

Unified Heartbeat 是你的个人项目管理自动循环。它从你的视角出发，自动扫描你参与的所有 Concordia 项目，判断你作为 Owner 或 Member 应该做什么，然后在白名单允许的范围内替你执行一个任务。

它不是 Project Agent 的心跳（那是从项目管理者视角），也不是 Member Agent 的心跳（那是从成员视角）。它是唯一一个从你个人视角、跨所有项目运行的循环。

你的所有项目 → 扫描 7 个信源 → 按角色决策 → 白名单门控 → 执行 1 个任务 → 写日志

👥四种角色

系统中有且只有四种角色，从不混用"owner"这个歧义词：

👤

Project Owner

项目的负责人（人类），拥有项目

🤖

Project Agent

项目的 AI 代理（PMO Agent）

👤

Member

项目成员（人类），做具体工作

🤖

Member Agent

成员的 AI 代理

Unified Heartbeat 为你运行——你在某个项目中是 Owner 就走 Owner 分支，是 Member 就走 Member 分支。如果两者兼任，Owner 优先。

⚡执行流程

每次心跳对每个项目独立执行以下 7 步：

0

枚举项目

列出你参与的所有活跃项目，判断你在每个项目中的角色（Owner / Member）

1

读取上下文

加载 PROJECT.md、你的 todos、最近 2 次心跳日志。首次运行时，Tier 1 信源回溯 2 周，Tier 2 SharePoint 回溯 6 个月

2

扫描 7 个信源（子 Agent）

每个信源一个独立子 Agent，各自有充裕的 token 空间。Tier 1 每次心跳跑，Tier 2 每天第一次跑

3

按角色决策

Owner 分支 → 管理者思维（分配、跟进、review）；Member 分支 → 执行者思维（理解任务、深度思考、产出交付物）

4

执行 1 个任务（白名单门控）

选出最高优先级的一个任务，白名单检查通过后，用专属子 Agent 深度执行

5

写心跳日志

完整记录扫描结果、思考过程、执行动作、collection proposals、待人类决策的事项

6

更新 Todos

持久化你在这个项目中的任务清单——状态变更、新增项、交付物位置、备注

📡7 个信源

每个项目的信息扫描由独立子 Agent 执行——一个信源一个子 Agent，确保不会因为 token 不够而跳过。

子 Agent信源频率扫描内容
2a Project Channel 每次 项目专属 Teams 频道的新消息
2b Teams 聊天 每次 1:1 和群聊中与项目相关的对话 + 外部 URL fetch
2c 日历 每次 新增/变更的日历事件
2d 邮件 每次 收发邮件中与项目相关的内容 + 外部 URL fetch
2e 会议纪要 每次 会议 transcript / recap
2f SharePoint 文档 每天 项目 SP 站点中新增/修改的文档（动态关键词）
2g 公开网络 每天 主动搜索竞品、行业、技术动态（动态关键词）

动态关键词：2f 和 2g 的搜索关键词不是固定的——Agent 根据项目当前状态（PROJECT.md + todos + 近期日志）临时生成，确保搜索方向与项目最新需要一致。

🔀角色分支

Branch O — Project Owner

管理者思维：让整个项目往前走

  • 未分配的任务 → 决定方向、分配给 Member
  • 进行中的任务 → 跟踪进度、识别风险
  • 已提交的交付物 → 按验收标准 review
  • 风险/阻塞 → 记录并上报

Branch M — Member

执行者思维：深度理解和执行任务

  • 先扫描 Owner 最近动态获取上下文
  • 按优先级选一个任务深度执行
  • 遇到歧义 → 标记 blocked，不猜
  • 产出交付物 + _submission.md

🛡️白名单机制

每个用户在每个项目中有一个 white-list.<user>.md，定义心跳可以自动做什么。

✅ Category 1 — 自己的工作

  • 所有文件读写操作（思考、笔记、中间产物）
  • 更新 todo、写日志、记录 proposals
  • 唯一限制：不能写入 deliverables/ 文件夹（交付物需要人类审批）

🚫 Category 2 — 与他人沟通

  • ✅ 可以：回复简单的事实状态查询（"项目进度如何？"→ 复述已有记录）
  • ✅ 可以：轻量确认（"收到，我看下"）
  • 🚫 不可以：承诺、决策、审批、表达观点
  • 🚫 不可以：主动联系他人、分享未记录的信息
  • 🚫 不可以：发任何需要用户先看一眼的内容

拿不准的时候 → 不做，记到日志里等 Review 时人类决策。

📁项目知识结构

每个项目在 SharePoint 上的 Project Repo 布局：

Project repo/
├── state/
│   ├── PROJECT.md          项目定义
│   ├── todos.<user>.md    个人 todo
│   ├── white-list.<user>.md 白名单
│   └── planning/
├── sources/
│   ├── owner-global/      内部讨论/会议/邮件
│   ├── sharepoint/        SP 文档
│   └── web/               公开网络资源
├── deliverables/
│   └── <task-id>/         交付物 + _submission.md
├── learnings/
└── logs/
    ├── heartbeats/        uh-YYYY-MM-DD_HH-MM.md
    ├── reviews/           人类 review 记录
    └── decisions/

🧭设计原则

一人一循环，跨所有项目 从你的视角扫描所有项目，Owner 和 Member 角色用不同分支处理
每项目每拍一个任务 深度优于广度——用专属子 Agent 给足 token 做高质量执行
心跳期间不打扰人 所有需要人类决策的事记到日志，由 Review Skill 统一呈现
提案不自动归档 收集到的素材只做 proposal，进 sources/ 需要人类审批（隐私+相关性门控）
白名单兜底 只有白名单明确允许的操作才自动执行，拿不准就不做
先想再做 执行前深度思考目的、验收标准、风险；遇到歧义标记 blocked 而不是猜
日志自去重 读上次日志避免重复提案和重复执行
完整可追溯 每次心跳一份日志、每个交付物一份 submission，全链路可回溯

🔗配套 Skill: Unified Review

Unified Heartbeat 产出日志和 proposals → Unified Review 把它们变成和你的自然对话：

💬

一次一个决策

带建议、原因、风险，等你回复再给下一个

⚡

确认即执行

你说"收"就写入 sources，说"都收"就批量处理

📋

Review Log

记录每个决策和执行结果，形成 heartbeat ↔ review 闭环
