Project Repo

每个 Concordia 项目的知识空间结构——信息怎么进来、怎么流转、怎么变成团队共识。

整体结构

Project Repo 是一个项目的所有知识和状态的存放地。它不是一个被动的文件夹，而是一个有明确流转逻辑的系统：信息从外部进入 sources，经过 AI 加工和人类审批，变成项目的共享认知（state/cognition），最终产出交付物（deliverables）。

Project repo/ ├── state/ 项目的大脑——认知状态、信号指令、成员立场 ├── sources/ 进入项目的原始素材（经人工审批后才入库） ├── drafts/ AI 自动产出的草稿（等人审批后才变成正式交付物） ├── deliverables/ 正式交付物（人工审批后从 drafts 搬入） ├── logs/ 所有自动化行为的日志记录 └── learnings/ 项目中积累的经验教训

state/ — 项目的大脑

这是整个系统最核心的部分。它记录了：项目是什么（PROJECT.md）、该关注什么（signal-mandate.md）、项目的共享认知状态（cognition/）、以及等待 DRI 决策的问题（pending-review/）。

PROJECT.md
项目定义文档。包含目标、交付物、成功标准、deadline、团队。是整个项目的"宪法"。
signal-mandate.md
信号指令——DRI 和 AI 对话后定义的"项目该关注什么"。包括：关注哪些方向、扫哪些信源、什么算高价值、什么是噪音、什么情况需要立刻通知 DRI。是 AI 自动扫描的行动纲领。

cognition/ — 认知状态子系统

Project Alignment Graph 的文件化表达——追踪项目的共享认知、每个成员的理解状态、以及两者的差距。

shared-state.md 项目真相
经 DRI 确认的项目真相。列出所有 accepted 的认知对象和状态。相当于"项目现在认为是真的东西"。
objects/ 结论
认知对象——项目认为是真的事（目标、决策、风险、假设等）。每个对象独立一个文件，由一个或多个 signal 支撑，经 DRI 确认后才成立。
状态流转：candidate → accepted / rejected / monitoring
signals/ 证据
信号——外部发生的事（邮件、会议、论文、Teams 消息）。它本身不是项目真相，只是可能影响真相的证据。Signal 是证据，Object 是结论。
candidates.md
候选信号——AI 扫描发现的、还没被 DRI 判断的信号。每条含来源、内容、AI 推断的影响、优先级。
admitted/
已准入信号——DRI 确认后从 candidates 移入，成为认知对象的支撑证据。
stances/ 成员立场
每个团队成员一个文件，记录 AI 推断的（或经本人确认的）对项目方向的理解。
状态流转：inferred（AI 推断）→ aligned（确认一致）/ misaligned（确认有分歧）
gaps.md 护城河
认知差距图——自动对比 shared-state 和每个 stance，列出谁的理解和项目真相不一致。这是 Concordia 最有防御性的层——平台能做摘要，但"知道谁认知落后"是独有能力。

pending-review/ — 等待 DRI 决策

queue.md
校准问题队列——AI 结合项目全貌生成的、给 DRI 回答的具体问题。每个问题都是"AI 的判断 + 建议 + 等你确认"的格式。DRI 打开驾驶舱看到的就是这个。

sources/ — 项目的输入素材

所有进入项目的参考资料都在这里。重要规则：素材不会自动入库——AI 扫描发现后只做提案，人工审批后才写入 sources/。这是隐私和相关性的门控。

owner-global/
来自内部沟通的素材——会议纪要、Teams 讨论、邮件中的关键内容。当前有 3 份邮件摘要（Dawei 认知设计、PMO skill 更新、Workspace 系统设计）。
sharepoint/
来自 SharePoint 文档库的素材——相关的设计文档、PRD、技术规范等。当前为空。
web/
来自公开网络的素材——竞品分析、技术论文、行业新闻。当前有 2 份 web 研究（DRI 认知契约、人机团队对齐）。

drafts/ — AI 的草稿区

AI 心跳自动产出的工作草稿暂存在这里。按成员分子文件夹。草稿不是正式交付物——需要人 review 批准后才搬到 deliverables/。这解决了一个问题：AI 不用等人来了才开始工作，它先做完存草稿，人只需要审批成品。

当前为空——项目刚建，尚未有心跳产出草稿。

deliverables/ — 正式交付物

经人工审批后的正式产出物。每个任务一个子文件夹，包含交付文件 + 一个 _submission.md 记录谁提交的、什么时候、摘要和修订历史。

当前为空——项目刚建。

logs/ — 自动化日志

所有自动运行的记录。如果出了问题，来这里看发生了什么。

heartbeats/
心跳日志——AI 每小时自动扫描一次，记录发现了什么、思考了什么、做了什么。当前有 16 条 Project Agent 心跳记录（6/30 全天）。
reviews/
Review 日志——每次人工 review 的决策记录（哪个问题、怎么判断、执行了什么操作）。当前 1 份（7/1 review）。
decisions/
重大决策日志。当前为空。
knowledge-fit/
知识健康度检查日志——AI 定期体检知识库的完整性、一致性、新鲜度。

learnings/ — 经验教训

项目执行过程中积累的经验和教训，分为执行类（怎么做事）和发现类（发现了什么）。用卡片形式记录，按时间索引。

当前为空模板——项目刚建。

信息怎么流转

整个系统的信息流是这样的：

外部信号（邮件/Teams/会议/Web） ↓ AI 按 signal-mandate 扫描 signals/candidates.md（候选信号，AI 推断了影响） ↓ DRI 在 Truth Gate 中判断 cognition/objects/（accepted = 项目真相更新） cognition/shared-state.md（索引更新） ↓ 自动对比 shared-state vs stances cognition/gaps.md（谁的理解落后了？） ↓ Loop 2: 问成员确认 stances/ 更新（aligned / misaligned / unknown）
AI 心跳发现需要做的事 ↓ 白名单允许？ drafts/（先存草稿） ↓ 人 review 批准 deliverables/（正式交付物）
