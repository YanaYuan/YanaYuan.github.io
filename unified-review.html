Unified Review

心跳的另一面——用自然对话的方式，帮你过一遍所有项目的待决策事项，确认即执行。

concordia-pmo-manager / skills / unified-review

💡这是什么

Unified Review 是 Unified Heartbeat 的人类面对面伙伴。心跳在后台自动扫描、思考、执行白名单内的工作；Review 把心跳产出的"需要你拍板"的内容，以自然对话的方式一个一个呈现给你，你说一句话就执行。

心跳（自动） → 发现任务 → 白名单允许？ → ✅ 直接执行
心跳（自动）→ 发现任务 → 白名单不允许？ → 写入日志等 Review

Review（你在场） → 呈现决策 → 你 approve？ → ✅ 当场执行
Review（你在场）→ 呈现决策 → 你 reject/defer？ → 记录，下一条

核心原则：approve 即执行。不存在"approve 了但留到下次心跳做"的中间态——你在场就是最高权限，没有理由延迟。

🎯如何触发

你随时可以说：

💬

"review"

或 "过一下"、"有什么需要我看的"

⏰

定时提醒

可配合 cron 每天某个时间提醒你 review

⚡执行流程

0

收集上下文（静默）

读所有项目的 PROJECT.md、你的 todos、Planner 任务、最新心跳日志和上次 review log。数出总决策点，按紧急度排序。你看不到这一步。

1

简报（每个项目开头）

2-4 句话告诉你：心跳发现了什么、有什么风险、接下来有几个决策点。纯 FYI，不需要你做任何事。

2

逐条呈现决策

Phase A（资料收集）→ Phase B（执行决策），一次一个，带建议、理由、风险、进度 [N/Total]。建议不收的自动跳过。

3

你回复 → 立即执行 + 立即写 log

每个决策完成后立刻执行、立刻持久化。即使 session 中途断了，已做的决策不丢。

4

跨项目切换

一个项目过完 → 一句话收尾 → 自然过渡到下一个项目的简报。

5

总结收尾

所有项目过完，3-5 个 bullet 总结做了什么。内部 log 补个 summary。

📋两个阶段

每个项目内的决策按固定顺序呈现：

📂

Phase A — 资料收集

心跳扫到的相关素材，问你要不要收进项目资料库。快速 yes/no，节奏快。

建议不收的自动跳过——只呈现建议收的。

⚡

Phase B — 执行决策

心跳想做但白名单挡住的操作：创建任务、发消息给团队、更新状态等。需要你多想一下。

📖开始前读什么

Review 独立于心跳运行——即使不在同一个 session，也能获取完整上下文：

📄

PROJECT.md

项目目标、团队、deadline

✅

你的待办清单

当前任务和状态

📊

Planner 任务

已有任务、分配、进度

📝

心跳日志

上次 review 后的所有 uh- log

🔄

上次 Review log

避免重复问已决策的事

🔓执行权限

Review 时你在场 = 最高权限。心跳做不了的事，review 时都能做：

📂

收录资料

写入项目资料库（心跳不能自动 file）

📋

创建 Planner 任务

分配给团队成员

💬

给团队发消息

经你确认内容后发送

📦

提交交付物

写入 deliverables/

💾即时持久化

每个决策完成后立即写入 review log + 更新状态，不等到最后批量写。

🛡️

断点安全

session 中途断了也不丢已做的决策

🔄

心跳闭环

已 review 的 proposal 标记完成，下次心跳不重复提

⏸️

Defer 也安全

"以后再说"的写入待办，心跳通过待办就知道不用再提

🎨对话风格

✓ DO

  • 像同事给 brief，不像系统报告
  • 用业务语言（"项目资料库"、"项目频道"）
  • 每条都给建议 + 理由
  • 支持"都收"批量操作
  • 附原链接和参与人
  • 允许打断、提问、换方向

✗ DON'T

  • 暴露文件路径、ID、内部结构
  • 一次给多个决策
  • 倒一大堆内容摘要
  • 问"先看哪个项目？"
  • 说"根据心跳日志分析..."
  • 告诉用户"我在写 log"

🧭设计原则

简报在前 先给上下文再问决策，让用户知道全貌
一次一个决策 不 overwhelm；想快可以说"都收"
资料在前，执行在后 快决策（yes/no）先过完，再处理需要思考的
永远有建议 agent 有观点，不是中立陈列选项
链接优先 用户想看细节点链接，agent 只说为什么
业务语言 用户永远看不到文件路径和系统 ID
进度可见 [N/Total] 让用户随时知道还剩多少
approve 即执行 不存在"approve 了但留到下次做"的情况

🔗与 Unified Heartbeat 的关系

Heartbeat（自动、静默、每小时）
扫描 7 信源 → 发现任务 → 白名单内直接做 → 白名单外写日志

↓ 产出日志 + proposals + todos ↓

Review（手动触发、对话式、你在场）
读日志 → 呈现待决策 → 你 approve → 当场执行 → 写 review log

↓ 更新 todos + 标记已处理 ↓

下次 Heartbeat 读 todos → 知道什么已处理 → 不重复
